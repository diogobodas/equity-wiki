# Wiki Schema

This file defines the structure, conventions, and workflows for this wiki.
The LLM agent reads this file before every wiki operation.

## Purpose

Two complementary uses:
1. **Knowledge wiki** — entity, concept, sector, comparison and synthesis pages about Brazilian listed companies, with citations.
2. **Modeling substrate** — preserved, structured source material that an LLM can use to build/maintain detailed financial models (planilha completa) for those companies. The wiki and the modeling layer live side by side: the wiki carries the *thesis*, the modeling layer carries the *numbers and the qualitative footing* (notas explicativas, MD&A).

## Architecture — five layers

```
sources/
├── undigested/                                   # inbox — raw files land here
├── full/{empresa}/{periodo}/{tipo}.md            # structured-but-uncut transcription (LLM floor)
├── structured/
│   ├── _schemas/{setor}.json                     # canonical schema per sector (created on demand)
│   └── {empresa}/{periodo}/{tipo}.json           # canonical + company_specific data
├── digested/{name}_summary.md                    # wiki-facing TL;DR
├── manifests/{empresa}.json                      # per-empresa discovery manifest (coverage, sources, precedence)
├── index.md                                      # registry of all sources
└── notion_tracker.md                             # Notion ingest state
```

Wiki pages (`*.md` at repo root) are the synthesis layer on top of all of the above.

### What each layer is for

| Layer        | Lossiness | Purpose | Generated by |
|--------------|-----------|---------|--------------|
| `undigested/` | 0%       | Inbox only — deleted after ingest | User |
| `full/`      | ~content-lossless (loses only PDF layout/images) | The "floor" — LLM reads this when modeling, re-extracting, or verifying a claim. **Replaces keeping the raw PDF.** | LLM during ingest |
| `structured/` | structured subset only | Feeds the planilha. Cross-empresa comparable. Determinístico. | LLM during ingest, only for ITR/DFP/release |
| `digested/`  | high — editorial extract | Wiki-facing TL;DR, used to write entity pages | LLM during ingest |
| `manifests/` | index-only  | **Discovery layer.** One JSON per empresa listing available periods, coverage per canonical block, source inventory, precedence rules and caveats. **A cold-start modeling agent reads this file FIRST** to know what data exists and where to find it. | LLM at end of every heavy/light ingest |
| Wiki pages   | narrative | Thesis, comparisons, concepts. Cite the layers above. | LLM, ongoing |

**Rule:** once a source is ingested, the original in `undigested/` is deleted. `full/` is the new floor — there is no `raw/`.

## The `full/` layer

A `full/{empresa}/{periodo}/{tipo}.md` file is a **structured but uncut** markdown transcription of one source.

- **Structured:** organized in headings — for an ITR/DFP, the canonical order is: `## Demonstração do Resultado`, `## Balanço Patrimonial`, `## Fluxo de Caixa`, `## DMPL`, `## DVA`, `## Nota 1 — ...`, `## Nota 2 — ...`, ..., `## Relatório da Administração / MD&A`. For other source types (apresentação, fato relevante, release), use the source's own structure.
- **Uncut:** within each section, content is **literal**. Tables are remounted in markdown. Numbers, footnotes, references — nothing is summarized, nothing is dropped. The LLM only reorganizes; it does not editorialize.
- **One source = one file.** Notes do not become separate files (unless promoted — see `promote_nota`).
- **Images, charts, layout decoration are dropped.** Captions and chart underlying numbers (when present in the source) are kept as text/tables.

## The `structured/` layer

Only generated for sources that contain canonical financial statements: ITR, DFP, earnings releases.

### File shape

```json
{
  "_schema": "banco/v1",
  "_schema_path": "sources/structured/_schemas/banco.json",
  "_empresa": "itau",
  "_periodo": "3T25",
  "_source": "full/itau/3T25/itr.md",
  "canonical": {
    "dre": { "receita_juros": null, "despesa_juros": null, "margem_financeira": null },
    "bp":  {},
    "fc":  {},
    "segmentos": {}
  },
  "company_specific": {
    "custo_credito_por_linha": {},
    "segmentos_gerenciais": {}
  }
}
```

- **`_schema`** is the sector schema identifier with version (`banco/v1`, `incorporadora/v1`, ...).
- **`_schema_path`** is the filesystem path to the schema file. **Mandatory** — a reader should never have to infer the schema location; the file itself points to it.
- **`canonical`** follows `_schemas/{setor}.json`. Comparable cross-empresa. This is what the planilha pulls from.
- **`company_specific`** is free-form. Captures whatever the company reports its own way (gerencial breakdowns, custo de crédito por linha, métricas operacionais idiossincráticas). No schema.
- Each numeric value is the number *as reported*. Conversions/normalizations belong to the modeling layer, not `structured/`.

### Schemas setoriais (`_schemas/{setor}.json`)

- Created **on demand** — the first banco ingested creates `_schemas/banco.json`; the first incorporadora release creates `_schemas/incorporadora.json`; and so on.
- Defines **only the `canonical`** — standardized keys every player in the sector fills.
- Grows over time: when a new key becomes recurrent across multiple players, it is promoted from `company_specific` into the sector's `canonical`. This promotion is a conscious operation, logged in `log.md`.
- Existing schemas are versioned (`banco/v1`, `banco/v2`) on breaking changes. Backwards-compatible additions do not bump the version.

## The `manifests/` layer

One JSON file per empresa at `sources/manifests/{empresa}.json`. **Purpose: discovery.** A cold-start modeling agent that wants to model `{empresa}` reads this file FIRST to know: which periods exist, which canonical blocks are filled in each period, which source files to consult, how to resolve conflicts between sources, and what caveats apply.

The manifest is **generated by the LLM at the end of every heavy/light ingest** that touches structured data. Ingests of web/notion sources do not affect manifests.

### Manifest shape

```json
{
  "_schema": "manifest/v1",
  "_updated": "2026-04-08",

  "empresa": "tenda",
  "display_name": "Construtora Tenda S.A.",
  "aliases": ["Tenda", "TEND3", "Alea"],
  "ticker": "TEND3",
  "setor": "incorporadora",
  "setor_schema": "sources/structured/_schemas/incorporadora.json",
  "segmentos": ["tenda_mcmv", "alea_sbpe"],
  "wiki_page": "tenda.md",

  "coverage": {
    "1T26": {
      "operacional":         {"status": "filled", "source": "sources/structured/tenda/1T26/data_pack.json"},
      "dre":                 {"status": "empty",  "source": "sources/structured/tenda/1T26/data_pack.json", "reason": "release completo do 1T26 ainda não publicado"},
      "bp":                  {"status": "empty",  "source": "sources/structured/tenda/1T26/data_pack.json", "reason": "idem"},
      "financeiro_ajustado": {"status": "empty",  "source": "sources/structured/tenda/1T26/data_pack.json", "reason": "idem"}
    },
    "4T25": {
      "operacional":         {"status": "filled", "source": "sources/structured/tenda/4T25/data_pack.json"},
      "dre":                 {"status": "filled", "source": "sources/structured/tenda/4T25/data_pack.json"},
      "bp":                  {"status": "filled", "source": "sources/structured/tenda/4T25/data_pack.json"},
      "financeiro_ajustado": {"status": "filled", "source": "sources/structured/tenda/4T25/data_pack.json"}
    }
  },

  "sources": [
    {
      "type": "previa_operacional",
      "asof": "1T26",
      "ingested_on": "2026-04-08",
      "full": "sources/full/tenda/1T26/previa_operacional.md",
      "digested": "sources/digested/tenda_previa_operacional_1T26_summary.md",
      "structured": ["sources/structured/tenda/1T26/previa_operacional.json"]
    },
    {
      "type": "data_pack",
      "asof": "1T26",
      "ingested_on": "2026-04-08",
      "full": "sources/full/tenda/data_pack_1T26.md",
      "digested": "sources/digested/tenda_data_pack_1T26_summary.md",
      "structured_glob": "sources/structured/tenda/{period}/data_pack.json",
      "structured_periods": ["1T26", "2025", "4T25", "3T25", "..."]
    }
  ],

  "precedence": [
    {
      "scope": "operacional.* at 1T26",
      "winner": "sources/structured/tenda/1T26/previa_operacional.json",
      "reason": "prévia é mais curada e detalhada; data_pack é redundante no 1T26 (números idênticos, cross-validado)"
    },
    {
      "scope": "dre.* | bp.* | financeiro_ajustado.* at any period",
      "winner": "sources/structured/tenda/{period}/data_pack.json",
      "reason": "prévia operacional não traz DFs"
    }
  ],

  "caveats": [
    "Séries pré-2024 refletem empresa estruturalmente diferente (ciclo de quase-quebra 2021–2023).",
    "Alea (segmento SBPE) só tem histórico a partir de ~2020."
  ],

  "related_digests": [
    "sources/digested/tenda_previa_operacional_1T26_summary.md",
    "sources/digested/tenda_data_pack_1T26_summary.md"
  ]
}
```

### Field semantics

- **`coverage`** is a flat map `{period: {block: {status, source, reason?}}}`. Periods mix quarterly codes (`1T26`) and annual codes (`2025`) — consumers distinguish by format.
- **`status`** is one of:
  - `"filled"` — the referenced structured file has this block populated (at least one leaf non-null).
  - `"empty"` — the structured file exists but this block is all-null; `reason` should explain why (release not yet published, etc).
  - `"partial"` — some leaves filled, some null; use `reason` to clarify.
  - `"na"` — this block does not apply to this empresa/period (e.g., a segment that did not exist yet).
- **`sources`** is an array of source file entries, ordered by ingestion. Each has `type` (matching the source type tokens), `asof` period, ingestion date, and paths to each layer's artifacts. For sources that produced many structured files (data packs), use `structured_glob` + `structured_periods` instead of listing every path.
- **`precedence`** is an ordered list of rules, each with a `scope` string (human-readable, not strictly machine-parsed), a `winner` path, and a `reason`. Used when two or more sources cover the same field+period and the agent needs to pick one.
- **`caveats`** are short strings — non-trivial facts a modeler needs to know before trusting the data (structural breaks, ongoing events, definitional quirks).
- **`related_digests`** points at the digested summaries, for context-rich reading when needed.

### Manifest maintenance (mandatory in every heavy/light ingest)

At the end of every `file` ingest (heavy or light) for an empresa, the LLM MUST:

1. Read `sources/manifests/{empresa}.json` if it exists; otherwise prepare a new one with empresa metadata.
2. Update `_updated` to today's date.
3. Add or update the `sources` entry for this ingest — idempotent by `(type, asof)`.
4. For every period touched by this ingest, update `coverage[period][block].status` and `.source`. Compute `status` by inspecting the written structured file: leaves all-null → `empty`, all non-null → `filled`, mixed → `partial`.
5. Add/update `precedence` rules if this ingest creates a new conflict with an existing source.
6. Add `caveats` entries if the ingest revealed non-trivial structural facts (optional but encouraged).
7. Validate: every path referenced in `coverage[*].source` and in `sources[*]` actually exists on disk.
8. Write the manifest back, overwriting in place.

First ingest of a new empresa creates the manifest from scratch.

## Page Types (wiki layer)

| Type | Description | Naming Example |
|------|-------------|----------------|
| entity | A company, person, product, or organization | `itau.md`, `cyrela.md` |
| concept | A technique, metric, principle, or idea | `custo_risco.md`, `nim.md` |
| sector | An industry or market segment | `bancos.md`, `incorporadoras.md` |
| comparison | Side-by-side analysis of 2+ entities/concepts | `itau_vs_bradesco.md` |
| synthesis | Cross-cutting analysis or thesis | `tese_selic_alta.md` |
| nota | Promoted nota explicativa (see `promote_nota`) | `itau_nota_instrumentos_financeiros.md` |

## Frontmatter (mandatory)

```yaml
---
type: entity | concept | sector | comparison | synthesis | nota
aliases: [Alternative Name, TICKER]            # optional
sources: [full/itau/3T25/itr.md, ...]          # paths into sources/
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

## Naming Conventions

- Filenames: `snake_case.md`, lowercase, Portuguese when natural
- No ticker prefixes on wiki pages — tickers go in `aliases`
- Period codes in `full/` / `structured/`: `1T25`, `2T25`, `3T25`, `4T25`, `2025` (annual)
- Source type tokens in filenames: `itr`, `dfp`, `release`, `previa_operacional`, `data_pack`, `apresentacao`, `fato_relevante`, `call_transcript`. The `data_pack` token designates a RI-maintained historical spreadsheet spanning many periods; path is `full/{empresa}/data_pack_{asof_period}.md` (no per-period subfolder) because it is a single as-of document.

## Source Citations

Every factual claim must trace back to a source. Citation format depends on layer:

- **Qualitative / context:** `(fonte: full/itau/3T25/itr.md §nota_18)`
- **Numeric claim:** `(fonte: structured/itau/3T25/itr.json :: canonical.dre.margem_financeira)`
- **Legacy / digest-only:** `(fonte: digested/{name}_summary.md)` — only when the underlying source has no `full/` (notion notes, web sources).
- **Web:** `(fonte: url, confiabilidade: oficial|editorial|community)`

### Dated Claims

Claims that can become factually wrong over time without the period changing carry an explicit `em: YYYY-MM-DD` in the citation:

- `(fonte: digested/Texto Reforma tributaria jan-2025_summary.md, em: 2025-01-16)` — `em:` is the date the information is **true in the real world** (publication date of a law/MP/LC, announcement date of corporate guidance, effective date of a portaria). It is **not** the ingest date.
- Ingest date stays in `manifests/{empresa}.json :: sources[].ingested_on`. Do not replicate it on the claim.

| Claim type | Gets `em:`? | Example |
|---|---|---|
| Alíquota, regra fiscal, dispositivo de lei | yes | `em: 2025-01-16` (LC 214/2025) |
| Guidance corporativo forward-looking | yes | `em: 2026-04-10` |
| Valor regulatório (teto MCMV, faixa) | yes | `em: 2024-10-03` |
| Meta operacional datada (frota, capex) | yes | `em: 2026-03-15` |
| Número de DF por período (margem 3T25, ROE) | no | período já codifica a data |
| Definição conceitual ("CBS substitui PIS/Cofins") | no | atemporal |

Rule of thumb for the LLM when writing a claim: if the claim can be factually wrong tomorrow without the accounting period changing, it carries `em:`. If it is tied to an explicit accounting period or is definitional, it does not.

## Supersession

When new information updates a dated claim, the LLM chooses one of two modalities based on the nature of the change.

### Modality 1 — Silent overwrite (default)

For guidance refreshes, number updates, or effective date changes without regime changes. Old claim removed, new claim written, `em:` updated. History lives in git log.

```
antes: Alíquota reduzida em 50% (fonte: X, em: 2025-01-16)
depois: Alíquota reduzida em 60% (fonte: Y, em: 2026-08-10)
```

### Modality 2 — Structural supersession ("antes × depois")

For regime changes, law amendments with transition rules, or restructured tariffs. The LLM writes an inline table or section comparing the old and new states — because the contrast itself is the analytical content. See [[section_232]] for an example pattern — the "Revisão de abril/2026 — antes × depois" table.

**Triggers for Modality 2:**
1. The change invalidates an analytical premise (not just the number), OR
2. The old claim is referenced from other pages via wikilink/cross-cite (silent overwrite would break cross-page consistency).

**Not used:** strikethrough inline, HTML-comment history, separate "history" pages. Git log preserves evolution; the page stays readable.

## Wikilinks

- `[[page_name]]` between wiki pages, Obsidian-compatible.
- Only link to pages that exist or should exist.
- First mention in a section gets linked; subsequent mentions in the same section do not.
- Wikilinks are for the wiki layer only — `full/` and `structured/` files use plain paths, not wikilinks.

## Watchlist

Opt-in mechanism for pages with claims that can change outside the CVM/Notion/calls ingest pipeline (laws, regulations, corporate forward-looking statements not yet reported in a filing).

### Declaration

Frontmatter gains an optional `watches:` array:

```yaml
---
type: concept
aliases: [CBS, IBS, LC 214/2025]
watches:
  - query: "LC 214/2025 alteração alíquota"
    sites: [planalto.gov.br, mattosfilho.com.br]
    cadence: weekly
  - query: "reforma tributária incorporadora IBS CBS 2026"
    sites: [valor.globo.com, infomoney.com.br]
    cadence: monthly
sources: [...]
updated: 2026-04-22
---
```

Fields: `query` (string for WebSearch), `sites` (domain list restricting the search), `cadence` (`weekly` | `monthly` | `quarterly`).

### Flow

`tools/watch.sh` → `tools/lib/watch_runner.py` → reads pages with `watches:` → checks each entry against its cadence (via `sources/watch_state/{page_slug}.json`) → runs a WebSearch restricted to the `sites:` → diffs results against `known_urls` in state → emits `[watch-hit]` entries to the next lint report.

**Not auto-ingested.** The runner signals only. The human decides whether to pull the hit through `fetch_notion.sh` or `fetch.sh`.

## Quality Principles

- Never invent data — every number traces to `structured/` or `full/` (or to a registered web/notion source).
- Prefer specific over vague: "ROE de 23.4% em 2025" not "ROE alto".
- Flag contradictions explicitly rather than silently choosing.
- For dated claims (those carrying `em:` per §Dated Claims), supersession follows §Supersession — update `em:` or write an "antes × depois" block. Do NOT add `[stale: ...]` on top of `em:` claims.
- The `[stale: last verified YYYY-MM-DD]` marker remains available for claims that lack a citable source date (editorial annotation, legacy pages not yet migrated).
- **Never edit files in `sources/`** — sources are immutable. To correct, re-ingest.
- **`log.md` is append-only.**
- **One source at a time** during ingest, so cross-linking stays coherent.

| Type | Origin | Layers produced | Reliability |
|------|--------|-----------------|-------------|
| `file` (ITR/DFP/release) | PDF/XLSX in `undigested/` | `full/` + `structured/` + `digested/` | — |

| Level | Examples | Trust |
|-------|----------|-------|
| `oficial` | IR sites, BCB, CVM, ANBIMA, regulators | High — quantitative |
| `editorial` | Valor, InfoMoney, Bloomberg, Reuters | Medium — context |
| `community` | Wikipedia, blogs, forums | Low — conceptual only |

## PDF pre-processing

PDF sources in `undigested/` are pre-processed with [opendataloader-pdf](https://github.com/opendataloader-project/opendataloader-pdf) before the LLM ingest step. This eliminates extraction gaps in tables, DMPL, pareceres, and visual-heavy pages that the LLM struggles to read directly from PDF.

```bash
# Install (one-time)
pip install -U opendataloader-pdf

# Convert PDF to markdown (standard mode — Java required)
opendataloader-pdf <file>.pdf --format markdown --use-struct-tree --table-method cluster

# Optional: hybrid mode for image-heavy pages (OCR via docling)
# Requires: pip install -U "opendataloader-pdf[hybrid]" && opendataloader-pdf-hybrid --port 5002
# Then add: --hybrid docling-fast
```

- The output `.md` replaces the PDF as input for the LLM ingest steps below.
- Both the original PDF and the intermediate `.md` are deleted from `undigested/` after ingest completes (same lifecycle as today — `undigested/` is an inbox).
- Non-PDF sources (XLSX, notion, web) are unaffected.
- **Known limitation:** pages rendered as images in the PDF (e.g., some DMPL tables) require hybrid mode for extraction. In standard mode these come out as image references — the LLM should note the gap and fill manually where possible (cross-referencing BP/DRE for reconciliation).

## Wiki Operations

### Ingest (file — ITR/DFP/release)

The "heavy" path. Runs one source at a time.

0. **If source is PDF**, run `opendataloader-pdf <file>.pdf --format markdown --use-struct-tree --table-method cluster` in `undigested/`. Use the resulting `.md` as input for steps 1–9.
1. Read the source fully from `sources/undigested/`.
2. **Generate `full/{empresa}/{periodo}/{tipo}.md`** — structured-but-uncut transcription. Headings as defined above. Tables in markdown. Notas explicativas as `## Nota N — título` headings, literal content.
3. **Generate `structured/{empresa}/{periodo}/{tipo}.json`**:
   - If `_schemas/{setor}.json` does not exist yet, create it from this source (canonical keys = whatever the source reports for the standard DFs).
   - Fill `canonical` following the schema. Missing keys → `null`, never omit.
   - Put gerencial / idiosyncratic data in `company_specific`.
4. **Generate/update `digested/{name}_summary.md`** — wiki-facing TL;DR.
5. **Update wiki pages** (`itau.md`, sector page, relevant concept pages) with new claims and citations pointing into `full/` or `structured/`.
6. **Update `sources/manifests/{empresa}.json`** — add/update the source entry, update coverage for every period touched, refresh precedence and caveats. See "Manifest maintenance" above.
7. **Delete** the original from `undigested/`.
8. Update `sources/index.md` (status = digested, paths to all layers produced).
9. Append entry to `log.md`.

### Ingest (file — apresentação / fato relevante / outros)

The "light" path — no `structured/`.

0. **If source is PDF**, run `opendataloader-pdf` as described in "PDF pre-processing" above. Use the resulting `.md` as input.
1. Read source from `undigested/`.
2. Generate `full/{empresa}/{periodo}/{tipo}.md` — structured-but-uncut.
3. Generate `digested/{name}_summary.md`.
4. Update wiki pages.
5. Update `sources/manifests/{empresa}.json` — add the source entry under `sources` (no coverage change since no structured/ is produced).
6. Delete original. Update `sources/index.md` and `log.md`.

### Ingest (file — data_pack, update)

Data packs are sourced from RI-maintained XLSX spreadsheets that contain the full historical series plus the latest quarter. When a new spreadsheet arrives (e.g. as-of 2T26 replacing the prior as-of 1T26), the **entire spreadsheet is re-processed** — NOT just the new quarter. The spreadsheet is self-contained: it repeats all historical data by design.

0. Convert XLSX → markdown via ephemeral script (openpyxl). The script re-processes ALL sheets/periods in the spreadsheet.
1. Create `full/{empresa}/data_pack_{new_asof}.md`. The prior `full/{empresa}/data_pack_{old_asof}.md` is **NOT deleted** — it stays as a historical snapshot of what the RI reported at that point in time.
2. **Delta detection** (BEFORE overwriting structured/):
   For every period that already has a `structured/{empresa}/{period}/data_pack.json`:
   a. Compare each canonical and company_specific leaf value: new vs. existing.
   b. Flag as **restatement** if `|new − old| > 1%` of old OR `|new − old| > R$ 1 mm` (whichever threshold is hit first). Ignore `null→value` transitions (new data, not restatement).
   c. Log all restatements in `log.md` as a summary table: empresa, period, block.key, old_value, new_value, Δ%.
3. Write structured/:
   a. **New periods** (e.g. 2T26): create `structured/{empresa}/{period}/data_pack.json`.
   b. **Existing periods**: OVERWRITE with new values. Update `_source` and `_source_asof` to point to the new `full/` file.
   c. Exception: blocks already superseded by DFP/ITR (per manifest precedence) — still overwrite `data_pack.json` (it's the data_pack's own record), but the manifest precedence already ensures DFP/ITR wins for consumers.
4. Generate `digested/{empresa}_data_pack_{new_asof}_summary.md`. Include:
   - What's new (new periods, new blocks filled).
   - What changed (restatements detected in step 2).
   - Prior digest stays as historical record.
5. Update manifest:
   a. Replace the `sources` entry for `type=data_pack`: new `asof`, new `ingested_on`, updated `structured_periods`.
   b. Add coverage entries for new periods.
   c. Review precedence — typically unchanged (DFP/ITR still wins where it existed).
   d. Add caveats if restatements were material or structural.
6. Update wiki pages with new-period data.
7. Delete original XLSX from `undigested/`. Update `sources/index.md` and `log.md`.

### Ingest (web)

1. `WebSearch` with targeted queries (3–4 per topic).
2. `WebFetch` on the best results (max 3–5 URLs).
3. Classify reliability.
4. Update wiki pages with inline citations `(fonte: url, confiabilidade: nivel)`.
5. Register each URL in `sources/index.md` with type `web`.
6. Append to `log.md`.

No `full/` or `structured/` is generated for web sources — the URL itself is the floor.

### Ingest (notion)

1. Fetch via `mcp__claude_ai_Notion__notion-fetch` or discover via `notion-search`.
2. Extract facts, data, quotes.
3. Update wiki pages with `(fonte: notion, "page title")`.
4. Generate `digested/notion_{slug}.md`.
5. Update `sources/notion_tracker.md` and `sources/index.md`.
6. Append to `log.md`.

### Query

1. Search wiki pages first.
2. If the question is numeric/quantitative and points at a specific period, drop into `structured/` directly.
3. If the question is qualitative or needs context (notas, MD&A), open the relevant `full/` file.
4. Synthesize answer with citations.
5. If the answer is valuable and reusable, promote it to a new wiki page.
6. **Write-through backfill (N=1):** whenever step 3 happened — i.e., the query had to go to `full/` to fetch a *numeric or structurable* fact that was not already in `structured/` — the agent MUST, before responding, add that fact to `company_specific` of the relevant `structured/{empresa}/{periodo}/{tipo}.json` (creating nested keys as needed). The first query pays the cost; the second is O(1).
   - **Scope:** `company_specific` only. Promotion to `canonical` (sector schema) remains a conscious operation logged in `log.md` — never automatic.
   - **What qualifies:** numeric values, dated events, structured tables (e.g., a nota's breakdown). Pure qualitative prose stays in `full/`.
   - **Keying:** mirror the nota/section path. E.g., a lookup into Nota 12 for lease balance writes to `company_specific.notas.nota_12_arrendamento.saldo_final_consolidado`.
   - **Citation:** the new key should carry a `_fonte` sibling pointing to the `full/` section it came from, so provenance is preserved.
   - **No-op cases:** if the query was qualitative (definition, context, narrative), or if the fact is already in `canonical`/`company_specific`, skip the backfill.
   - **Log:** append a one-line entry to `log.md` under a `backfill` section noting empresa, periodo, key path, and source section.

### Modeling (planilha)

1. Identify the empresa and the period range.
2. Pull `canonical` series from `structured/{empresa}/{*}/{tipo}.json` for each period — this is the deterministic spine of the model.
3. Pull `company_specific` for gerencial breakdowns.
4. Open `full/` for notas explicativas and MD&A when grounding a projection premise.
5. Cross-check with the wiki page (`itau.md`, etc) to incorporate the qualitative thesis.
6. Log the modeling session in `log.md` (empresa, period, files consulted).

### `promote_nota`

When a nota explicativa is referenced from 3+ wiki pages, or the user explicitly requests it, promote it to its own wiki page.

1. Create `{empresa}_nota_{slug}.md` with `type: nota`.
2. Consolidate the nota across all available periods (pull each period's heading from the corresponding `full/`).
3. Add wikilinks from the entity page and from any other page that references the nota.
4. Future ingests touching this nota must update the promoted page.
5. Append to `log.md`.

### Lint

Periodic health check:

1. **Dead links** — `[[wikilinks]]` to non-existent pages.
2. **Orphan pages** — wiki pages with zero inbound links.
3. **Stale pages** — source newer than the wiki page that cites it.
4. **Missing cross-refs** — page mentions a concept that has a wiki page but doesn't link it.
5. **Contradictions** — same metric, different values across pages.
6. **Schema drift** — `structured/` files where `canonical` is missing keys defined in the current `_schemas/{setor}.json` version.
7. **Unpromoted recurring notas** — notas referenced from 3+ pages without a promoted page.
8. **Missing `_schema_path`** — any structured JSON that lacks the mandatory `_schema_path` field.
9. **Manifest staleness** — for each `sources/manifests/{empresa}.json`:
   - (a) *Stale*: any file under `sources/structured/{empresa}/` has mtime newer than the manifest's `_updated`.
   - (b) *Rotten*: the manifest references a path in `coverage[*].source` or `sources[*]` that does not exist on disk.
   - (c) *Missing*: a `sources/structured/{empresa}/` directory exists but no corresponding manifest file.
   - (d) *Coverage mismatch*: `coverage[period][block].status == "filled"` but the referenced structured file has that block all-null (or vice versa).
10. **Dated claim staleness** — parse `(fonte: X, em: YYYY-MM-DD)` across all pages and flag:
    - (a) *Age threshold exceeded*: `em_date + threshold < today` and no newer source on the same topic. Severity `warn`. Thresholds: legal/regulatório 12 months, guidance 6 months, metric absoluto 3 months (configurable via `tools/lint_config.json`).
    - (b) *Newer source available*: a `digested/`/`full/`/`structured/` file exists with `ingested_on > em_date` and matches the page's empresa/concept. Severity `action`.
    - (c) *Cross-page contradiction*: two pages with numerically conflicting claims on the same item, different `em:` dates. Severity `action`.
    - (d) *Missing `em:`*: claim contains a number plus a temporal verb (`vigente`, `a partir de`, `até`) but has no `em:`. Severity `hint`.
11. Report in `log.md`.
